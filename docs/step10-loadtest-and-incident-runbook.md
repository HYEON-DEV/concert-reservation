# Step10 부하테스트 및 장애 대응 매뉴얼

현재 시스템은 다음 구성 요소를 중심으로 동작한다고 가정한다.
- API Server: Spring Boot
- DB: MySQL
- Cache / Queue / Lock: Redis
- 비동기 후처리: Kafka

## 1. 문서 목적
이 문서는 최종 발표 자료가 아니라, 현재 구현된 콘서트 예약 서비스에 대해 실제로 부하테스트를 수행하고 장애가 발생했을 때 대응하기 위한 운영 매뉴얼이다.

범위는 두 가지다.
- 필수과제: 기본 부하테스트 진행 매뉴얼
- 선택과제: 장애 대응 매뉴얼

## 2. 현재 서비스의 핵심 장애 포인트
현재 구조에서 장애를 유발할 가능성이 높은 지점은 아래와 같다.

### 2.1 API / 애플리케이션
- 대기열 진입 요청 폭증
- 좌석 조회 캐시 미스 증가로 인한 DB 조회 폭증
- 결제 시 동시성 충돌 증가
- 처리되지 않은 예외로 인한 5xx 증가
- 긴 트랜잭션으로 인한 스레드 점유 증가

### 2.2 MySQL
- `findForUpdate` 기반 좌석 잠금 경합 증가
- 커넥션 풀 고갈
- 슬로우 쿼리 증가
- 임시배정 만료 정리와 결제/예약이 동시에 몰리며 write 부하 증가

### 2.3 Redis
- 대기열 토큰 발급 및 상태조회 집중
- 좌석 hold 키 증가
- 캐시 만료 시점에 동시 재조회 발생
- Redis 장애 시 대기열/캐시/분산락 기능 전체 영향

### 2.4 Kafka
- 예약 완료 메시지 publish 실패
- consumer lag 증가
- consumer 예외로 메시지 적체
- at-least-once 처리로 인한 중복 소비 가능성

## 3. 기본 부하테스트 진행 매뉴얼

## 3.1 부하테스트 목적
이번 서비스에서 부하테스트의 목적은 단순 TPS 측정이 아니라 아래 세 가지를 검증하는 것이다.
- 목표 TPS에서 서비스가 정상 응답하는지
- 평균/중간/최대 및 p95/p99 응답시간이 허용 범위인지
- 동시 요청 시 정합성 문제가 발생하지 않는지

## 3.2 우선 테스트 대상
우선순위는 실제 장애 가능성과 병목 강도를 기준으로 정한다.

### P1
- 대기열 토큰 발급 API
- 대기열 상태 조회 API
- 좌석 조회 API
- 좌석 예약 API
- 결제 API

### P2
- 포인트 충전 API
- 랭킹 조회 API
- 공연 목록/회차 조회 API

## 3.3 왜 이 구간을 테스트해야 하는가
### 대기열 토큰 발급 / 상태 조회
- 오픈 직후 가장 먼저 몰리는 구간이다.
- Redis 기반이라 빠르지만, 요청량이 급증하면 애플리케이션 CPU와 Redis 네트워크 병목이 먼저 드러난다.

### 좌석 조회
- 캐시 히트율이 낮으면 DB read 부하가 급격히 증가한다.
- 캐시 TTL이 짧기 때문에 특정 시점에 캐시 미스가 몰릴 가능성이 있다.

### 좌석 예약
- 같은 좌석을 여러 사용자가 동시에 점유하려는 경쟁 상황이 발생한다.
- Redis hold + DB 상태 변경이 함께 동작하므로 정합성/레이턴시를 동시에 봐야 한다.

### 결제
- 핵심 트랜잭션이다.
- 포인트 차감, 결제 이력 저장, 좌석 확정, 이벤트 발행까지 포함되어 있어 DB/Redis/Kafka 영향을 모두 받는다.

## 3.4 테스트 목표값
현재 로컬/과제 환경에서는 절대적인 수치보다 기준선 수립이 중요하다. 최초 목표는 아래처럼 둔다.

### 공통 기준
- 성공률: 99.5% 이상
- 5xx 비율: 0.5% 미만
- p95 응답시간
  - 조회 API: 300ms 이하
  - 예약/결제 API: 1000ms 이하
- 데이터 정합성
  - 동일 좌석 중복 예약 0건
  - 포인트 음수 잔액 0건
  - 결제 성공 후 좌석 상태 불일치 0건

### 대기열 기준
- 활성 사용자 한도는 현재 `100`
- 대기열 상태 조회 폭증 시에도 ACTIVE/WAITING 상태 계산이 깨지지 않아야 함

## 3.5 테스트 유형과 적용 순서
이번 과제에서는 모든 테스트를 다 하기보다 아래 순서로 진행한다.

### 1) Load Test
목표 부하에서 정상 처리 가능한지 확인한다.
- 예: 토큰 발급 200 RPS, 좌석 조회 150 RPS, 결제 30 RPS

### 2) Peak Test
짧은 시간 동안 급격히 부하를 높여 오픈 직후 트래픽을 가정한다.
- 예: 10초 동안 토큰 발급 1000 RPS burst

### 3) Stress Test
어디서부터 성능 저하와 오류가 시작되는지 임계점을 찾는다.
- 예: 100 RPS부터 100씩 올리며 실패율/latency 측정

Endurance Test는 시간이 길어 과제 범위에서는 선택으로 두되, Redis 메모리 누수나 Kafka lag 누적을 확인해야 할 때만 추가한다.

### 수집할 최소 지표
- API
  - RPS
  - 평균 응답시간
  - p95, p99
  - 4xx/5xx 비율
- JVM
  - CPU
  - Heap 사용량
  - GC 시간
  - 스레드 수
- DB
  - active connection 수
  - 커넥션 풀 사용률
  - slow query 여부
  - lock wait
- Redis
  - ops/sec
  - used_memory
  - evicted_keys
  - latency spike
- Kafka
  - producer error
  - consumer lag
  - 토픽 적체량

## 3.6 테스트 데이터 준비 기준
### 대기열 테스트
- 서로 다른 `userId` 1,000~10,000개 준비

### 좌석 조회/예약 테스트
- 특정 공연 1개
- 좌석 50~200개
- 사용자 500명 이상
- 같은 좌석에 경쟁을 유도할 테스트군과, 서로 다른 좌석을 예약하는 테스트군을 분리

### 결제 테스트
- 포인트가 충분히 충전된 사용자 세트 준비
- hold가 선행된 상태의 사용자 세트 준비
- 결제 성공/실패를 모두 확인할 수 있게 데이터 분리

## 3.7 기본 테스트 시나리오

### 시나리오 A. 대기열 진입 폭증
목적:
- Redis queue 처리량 확인
- ACTIVE/WAITING 계산 안정성 확인

절차:
1. 1,000명의 가상 사용자가 30초 동안 토큰 발급 요청
2. 이후 60초 동안 상태 조회 반복
3. ACTIVE 수가 100 근처에서 유지되는지 확인
4. EXPIRED 오판정, rank 꼬임 여부 확인

성공 기준:
- 토큰 발급 성공률 99% 이상
- 상태조회 p95 200ms 이하
- ACTIVE 100 초과 오차 없음

### 시나리오 B. 좌석 조회 집중
목적:
- 캐시 효율 확인
- 캐시 미스 시 DB read 부하 확인

절차:
1. 같은 공연의 좌석 조회 요청을 100~300 RPS로 5분간 수행
2. 캐시 warm-up 전/후 응답시간 비교
3. 캐시 TTL 만료 직전/직후 latency 스파이크 확인

성공 기준:
- warm-up 이후 p95 300ms 이하
- 캐시 만료 순간에도 5xx 급증 없음

### 시나리오 C. 동일 좌석 경쟁 예약
목적:
- 동시성 제어와 정합성 검증

절차:
1. 동일한 좌석 번호에 대해 50~100명의 사용자가 동시에 예약 시도
2. 성공 건수, 실패 건수, 최종 좌석 상태 확인

성공 기준:
- 성공 1건, 나머지 실패
- 좌석 상태가 HOLD 또는 RESERVED로 비정상 중복되지 않음

### 시나리오 D. 결제 집중
목적:
- 핵심 트랜잭션 병목 확인
- DB lock, Kafka 후처리 적체 여부 확인

절차:
1. hold가 된 좌석에 대해 결제 요청을 20~50 RPS로 수행
2. API 성공률, DB lock wait, Kafka lag 확인
3. 결제 성공 후 랭킹/데이터 플랫폼 후처리가 누락되지 않는지 확인

성공 기준:
- 결제 성공률 99% 이상
- p95 1000ms 이하
- Kafka consumer lag이 짧은 시간 내 해소

## 3.8 가정된 테스트 실행 결과
아래 결과는 로컬 과제 환경에서 테스트했다고 가정한 예시 결과다. 절대 성능 수치라기보다 병목 판단과 대응 근거를 남기기 위한 기준값이다.

### 결과 A. 대기열 진입 폭증
- 조건: 1,000 users / 30초 burst / 토큰 발급 + 상태조회
- 평균 RPS: 287
- 토큰 발급 성공률: 99.8%
- 상태조회 성공률: 100%
- 상태조회 p95: 118ms
- Redis used_memory: 42MB -> 58MB
- 오류: 일부 초기 요청에서 4xx 2건(중복 토큰 발급 시나리오)

판단:
- 현재 ACTIVE_LIMIT=100 정책은 정상 동작함
- Redis 자체 병목은 없었음
- 애플리케이션 CPU가 먼저 70% 근처까지 상승하여 API 서버 scale-out 여지가 더 큼

### 결과 B. 좌석 조회 집중
- 조건: 200 RPS / 5분 / 동일 performanceId 반복 조회
- warm-up 이전 p95: 412ms
- warm-up 이후 p95: 86ms
- 캐시 TTL 만료 직후 p95 spike: 244ms
- 5xx: 0건
- DB active connection 최대: 3/3

판단:
- 캐시가 성능 개선에 유효함
- 단, DB pool 크기가 3으로 작아 캐시 미스가 겹치면 즉시 병목이 발생함
- 운영 환경에서는 DB pool과 API replica 수를 함께 조정해야 함

### 결과 C. 동일 좌석 경쟁 예약
- 조건: 1개 좌석에 80명 동시 예약 시도
- 성공: 1건
- 실패: 79건
- 중복 HOLD/RESERVED: 0건
- p95: 624ms
- 예외 유형: `seat is not held`, `not seat holder`

판단:
- 정합성은 유지됨
- 동시성 제어는 기대대로 동작함
- 실패 응답 비율은 높지만, 이 시나리오는 비즈니스적으로 정상

### 결과 D. 결제 집중
- 조건: hold가 완료된 좌석 대상 30 RPS / 3분
- 성공률: 99.2%
- 5xx: 0.3%
- p95: 882ms
- DB lock wait 순간 증가: 최대 420ms
- Kafka lag 최대: 37 messages
- lag 해소 시간: 약 12초

판단:
- 목표 범위 내에서 처리 가능
- 병목은 DB write + lock wait
- Kafka는 적체가 발생하지만 짧은 시간 내 회복됨
- 데이터 플랫폼 전송을 API 응답과 분리한 구조는 유효함

## 3.9 테스트 중단 기준
아래 중 하나라도 충족하면 테스트를 중단하고 원인 분석으로 전환한다.
- 5xx 비율 5% 초과가 1분 이상 지속
- DB connection pool 100% 고갈
- Redis 응답 timeout 다발
- Kafka consumer lag이 지속 증가하며 회복되지 않음
- 애플리케이션 OOM 또는 restart 발생

## 3.10 테스트 후 분석 체크리스트
- 어느 API가 가장 먼저 병목이 되었는가
- 병목은 CPU, DB, Redis, Kafka 중 어디인가
- 정합성 오류가 있었는가
- 캐시 히트율이 예상과 맞았는가
- active queue 한도 100이 실제 정책과 맞는가
- Kafka 후처리 적체가 사용자 응답에 영향을 주는가
- 단기 개선으로 해결 가능한가, 구조 개선이 필요한가

## 3.11 테스트 결과 요약 및 결론
이번 가정 테스트 기준으로 가장 먼저 드러난 병목은 MySQL 커넥션 풀과 lock wait였다.

정리하면 다음과 같다.
- 대기열: Redis 기반 처리량은 충분하나 API 서버 CPU를 먼저 봐야 함
- 좌석 조회: 캐시 효과가 매우 큼
- 좌석 예약: 정합성은 확보됨
- 결제: 현재 구조에서 가장 비싼 구간이며 DB가 주요 병목

즉, 현재 시스템은 “정합성은 맞지만 고부하에서 DB가 먼저 제한 요소가 되는 구조”로 판단한다.

우선 개선 우선순위는 아래와 같다.
1. DB pool 및 쿼리 병목 개선
2. 좌석 조회 캐시 전략 고도화
3. Kafka consumer lag 관측 체계 강화

## 4. 장애 대응 매뉴얼

## 4.1 장애 대응 원칙
- 핵심 트랜잭션과 부가 기능을 분리해 영향 범위를 최소화한다.
- 장애가 발생해도 정합성을 우선 보장한다.
- 복구 전에는 증상 억제, 복구 후에는 재발 방지까지 포함한다.
- 로그만 보지 말고 메트릭, 에러율, 지연시간, backlog를 함께 본다.

## 4.2 장애 등급 정의
### Sev1
- 전체 예약/결제가 불가
- DB/Redis/Kafka 핵심 의존성 장애로 전체 서비스 중단
- 데이터 정합성 훼손 가능성 존재

### Sev2
- 일부 기능 장애
- 결제는 되지만 후처리 적체
- 대기열/좌석 조회 지연 급증

### Sev3
- 성능 저하 또는 비핵심 기능 장애
- 랭킹 지연, 데이터 플랫폼 전송 지연
- 운영자 개입이 필요하지만 서비스 핵심 기능은 유지

## 4.3 공통 장애 대응 프로세스
1. 장애 감지
2. 장애 분류 및 전파
3. 영향 범위 확인
4. 단기 완화 조치
5. 근본 원인 분석
6. 복구 확인
7. 장애 해소 공지
8. 사후 회고 및 액션 아이템 정리

## 4.4 공통 초기 확인 항목
장애 인지 후 5분 내 아래를 확인한다.
- 어느 API가 실패하는가
- 언제부터 시작됐는가
- 4xx/5xx 중 무엇이 증가했는가
- 응답지연인지, 완전 실패인지
- 단일 인스턴스 문제인지 전체 문제인지
- 최근 배포/설정변경/배치작업이 있었는지
- DB, Redis, Kafka 중 외부 의존성 장애가 있는지

## 4.6 시나리오별 장애 대응

### 시나리오 1. DB connection pool 고갈
징후:
- API timeout 증가
- DB connection active 수 최대치 도달
- 예약/결제 API부터 순차적으로 실패

즉시 조치:
1. 예약/결제 API latency 및 에러율 확인
2. DB active connection, lock wait, slow query 확인
3. 최근 대량 호출 API 또는 배치 중지
4. 필요시 읽기 비핵심 API를 임시 차단하거나 rate limit 적용
5. 병목 쿼리 확인 후 인덱스/쿼리 플랜 점검

추정 원인:
- 긴 트랜잭션
- `for update` 경합 증가
- 캐시 미스 폭증으로 DB read 급증
- 외부 I/O가 트랜잭션 안으로 들어온 경우

복구 확인:
- connection pool 사용률 정상화
- p95 회복
- 5xx 감소

후속 액션:
- 슬로우 쿼리 로그 수집
- 트랜잭션 범위 재검토
- 인덱스 추가 검토

### 시나리오 2. Redis 장애 또는 고지연
징후:
- 대기열 토큰 발급/조회 실패
- hold 검증 실패 급증
- 캐시 fallback으로 DB read 급증

즉시 조치:
1. Redis ping/latency 확인
2. 메모리 사용량, eviction, blocked clients 확인
3. 캐시성 기능과 핵심 기능 영향 분리
4. 필요시 대기열 진입 API를 fail-close 처리하거나 점검 모드로 전환
5. DB로 부하 전이되는지 즉시 확인

추정 원인:
- 메모리 부족
- 과도한 key 증가
- 네트워크 문제
- 단일 Redis SPOF

복구 확인:
- queue/hold/cache 관련 에러 감소
- DB read 부하 정상화

후속 액션:
- Redis HA 구성 검토
- 캐시 TTL/키 개수 상한 관리
- 대기열/hold 장애 시 graceful degradation 정책 문서화

### 시나리오 3. Kafka 적체 또는 consumer lag 증가
징후:
- 예약/결제는 성공하지만 후처리 지연
- 데이터 플랫폼 전송 누락처럼 보임
- lag 지속 증가

즉시 조치:
1. 토픽별 lag 확인
2. consumer 인스턴스 상태 확인
3. 최근 consumer 예외 로그 확인
4. 특정 메시지 반복 실패 여부 확인
5. 필요시 consumer scale-out 또는 문제 consumer 재기동

추정 원인:
- consumer 예외
- 외부 mock 데이터 플랫폼 호출 지연
- 파티션 대비 consumer 수 부족
- poison message 반복 처리

복구 확인:
- lag 감소 추세 확인
- 새 메시지 소비 지연이 허용 범위 내 회복

후속 액션:
- DLQ 도입
- 재시도 정책 분리
- consumer 멱등 처리 강화

### 시나리오 4. 중복 메시지 처리
징후:
- 동일 예약 완료 후처리가 2회 이상 수행
- 랭킹 점수 과다 증가
- 데이터 플랫폼 중복 전송

즉시 조치:
1. 동일 reservation/payment 기준 중복 로그 확인
2. at-least-once 소비 여부 확인
3. consumer ack 타이밍 확인
4. 영향 데이터 범위 산정

추정 원인:
- consumer 처리 후 ack 실패
- consumer 재시작
- producer 재전송

복구 확인:
- 신규 중복 발생 중단
- 보정 대상 데이터 목록 확보

후속 액션:
- idempotency key 저장
- processed message ledger 도입
- 상태 기반 중복 방지 로직 추가

### 시나리오 5. 좌석 중복 예약 또는 정합성 이상
징후:
- 동일 좌석이 둘 이상에게 예약된 흔적
- payment 기록과 좌석 상태 불일치

즉시 조치:
1. 해당 `performanceId`, `seatNo` 기준 데이터 조회
2. payment, seat, hold 상태를 함께 확인
3. 신규 같은 공연 예약을 임시 제한할지 판단
4. 영향 사용자 범위 식별

추정 원인:
- 동시성 제어 누락
- 예외 처리 중간 save/release 순서 문제
- 수동 데이터 수정

복구 확인:
- 해당 좌석의 최종 상태 정합성 확보
- 추가 중복 발생 없음

후속 액션:
- 경쟁 테스트 케이스 추가
- reserve/pay 시퀀스 재검토
- 보정 스크립트 준비

### 시나리오 6. 애플리케이션 OOM / 비정상 종료
징후:
- 프로세스 재시작 반복
- Heap 급증 후 다운
- GC 과다

즉시 조치:
1. 최근 요청 패턴 및 bulk 응답 확인
2. heap, GC, thread dump 확보
3. 대량 payload 응답이나 로그 폭증 여부 확인
4. 필요시 트래픽 제한

추정 원인:
- 캐시 과다 적재
- 대량 응답 객체 생성
- 무한 재시도/적체
- 비동기 backlog 누적

복구 확인:
- restart 종료
- heap 사용량 안정화

후속 액션:
- heap dump 분석
- 캐시 크기/TTL 재설계
- 대량 응답 paging 강제

## 4.9 장애 리뷰 예시
### 현상
- 발생 시각: 2026-03-03 20:14 KST
- 종료 시각: 2026-03-03 20:37 KST
- 영향 범위: 결제 API, 좌석 예약 API
- 고객 영향: 약 8.7% 사용자가 결제 지연 또는 실패 경험
- 비즈니스 영향: 오픈 20분 구간의 결제 전환율 저하

### 타임라인
- T0 20:14: 결제 API p95 3s 초과 알림 발생
- T1 20:17: DB pool 사용률 100%, lock wait 증가 확인
- T2 20:21: 비핵심 조회 요청 rate limit 적용, 정리 배치 중지
- T3 20:29: 응답시간 회복 시작
- T4 20:37: 에러율 정상화, 장애 해소 선언

### 원인
- 직접 원인: 동일 공연 결제 집중으로 좌석 row lock 경합 증가
- 근본 원인: MySQL pool 크기와 write 부하 대비 트랜잭션 처리 시간이 보수적으로 튜닝되지 않음

### 조치 내용
- 즉시 조치: 실시간 배치성 정리 작업 중지, 읽기 API 일부 rate limit 적용
- 임시 완화: 결제 요청 burst를 줄이기 위해 ingress 레벨에서 초당 요청 제한 적용
- 영구 조치: 좌석/결제 쓰기 경로 프로파일링 및 DB pool, 인덱스, 쿼리 재점검

### 5 Whys
1. 왜 장애가 발생했는가?
: 결제 요청이 특정 시간에 몰리며 DB lock wait가 급증했다.
2. 왜 lock wait가 급증했는가?
: 같은 performance에 대한 결제가 동시에 몰렸고 커넥션 풀이 작았다.
3. 왜 커넥션 풀이 작았는가?
: 로컬/과제 환경 설정을 운영 기준으로 재조정하지 않았다.
4. 왜 사전에 발견하지 못했는가?
: 결제 집중 시나리오 peak test를 먼저 하지 않았다.
5. 왜 재발 방지 장치가 없었는가?
: DB pool 사용률과 lock wait 기반 알림 임계치가 세밀하게 정의되지 않았다.

### 액션 아이템
- short-term: 결제 API peak test 추가, DB pool 사용률 알림 추가
- mid-term: 좌석/결제 write 경로 인덱스 재점검, lock wait 로그 수집 자동화
- long-term: 좌석 예약 트래픽 분산 전략과 replica/read-write 분리 검토

## 5. 현재 서비스 기준 권장 개선 항목
이번 과제 범위 밖이지만, 매뉴얼 관점에서 우선순위가 높은 개선은 아래와 같다.

### short-term
- API별 p95/p99, 5xx, DB pool, Redis latency, Kafka lag 대시보드 정의
- 예약/결제/대기열 에러 로그 구조화

### long-term

- 부하테스트 자동화 파이프라인 구축
